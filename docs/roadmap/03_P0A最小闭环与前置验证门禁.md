# P0A 最小闭环与前置验证门禁

## 结论

TS_RS 的近期目标不是把 RebarSmart 的 8 个 / 11 个钢筋命令全部铺开。

P0A 只证明一条完整链路能跑通：

```text
真实 STEP
  -> OCCT Viewer 水泥灰显示
  -> 选择边 / 面
  -> TopologyBinding 保存和恢复
  -> FixDistance 或 FixNumber 中心线生成
  -> RebarModel 保存
  -> 基础下料表
  -> 极简 Detail 包
  -> AutoCAD 插件 autoin 人工验证
```

## 为什么收缩 P0

外部技术评审指出：

```text
P0 如果同时追 8 个钢筋创建命令，
会变成每个命令都做一点，但没有一个生产闭环。
```

本项目接受该判断。

后续所有 P0 开发必须优先服务一个闭环，而不是优先扩按钮数量。

## P0A 必须前置的三个验证

### 1. TopologyBinding

必须先证明：

```text
同一个 STEP 多次导入，
关键 Face / Edge / Vertex 的 fingerprint 稳定。

保存 selectedFaceId / selectedEdgeId 后，
重开工程能找回对应拓扑对象。
```

没有这个验证，不允许把生成器绑定到真实选择结果。

### 2. Detail round-trip / 极简 autoin

必须先证明：

```text
旧 Detail 包能被 Reader 保守读取。
未知节点 / 未知属性不被丢弃。
Writer 能 round-trip 回写。
极简 Detail.xml + Detail01.stl XML 能被 CAD 插件 autoin 接受。
```

没有这个验证，不允许大规模设计 DrawingModel 字段。

### 3. RebarSmart 单位 / 枚举 / 默认值

必须先证明：

```text
UI 单位
内部单位
INI 单位
Detail 单位
默认值来源
枚举值来源
证据等级
GAP 状态
```

没有这个文档，不允许实现正式钢筋参数面板。

## FixDistance / FixNumber P0 命名边界

TODO-013 / TODO-014 已实现的是：

```text
FixDistanceCenterlineGenerator P0
FixNumberCenterlineGenerator P0
```

它们不是完整 RebarSmart 复刻。

P0 已做：

```text
基于 IGeometryEngine 的中心线生成。
覆盖定距 / 定数的主路径分布语义。
```

P0 未做：

```text
真实面投影。
真实导引面偏移。
过孔。
锚固。
完整弯钩。
复杂实体化。
完整 Detail 字段映射。
```

文档、todo 和后续评审里不得把当前 P0 说成完整 FixDistance / FixNumber。

## 禁止事项

```text
不要为了赶 UI，把 8 个钢筋按钮都做成可执行命令。
不要让 UI 面板直接解释 INI / CATNls / IDA 字段。
不要在 generator 内部 include OCCT 类型。
不要用 Detail 字段反向污染 RebarModel。
不要把旧项目【图石钢筋1比1复刻】里的钢筋业务代码迁入 TS_RS。
```

## 放行标准

进入真实 P0A 闭环开发前，必须满足：

```text
1. 老图石 Ribbon 外壳固定。
2. 老图石命令到 RebarSmart generator 映射表存在。
3. RibbonLite P0 实现边界存在。
4. TopologyBinding spike 计划存在。
5. Detail round-trip policy 存在。
6. RebarSmart 单位 / 枚举 / 默认值规则存在。
```
